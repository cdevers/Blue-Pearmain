# GH #99 — Reconcile convergence + Photos.app keyword write-back

**Issue:** `bp reconcile --fix` finds thousands of tag mismatches on every run and never converges.  
**Also:** Apple Photos ML labels should propagate back to Photos.app as explicit keywords for Smart Album use.

---

## Root causes

**Bug 1 — wrong source of truth in reconcile.**  
`check_photo()` compares `proposed_tags` against live Flickr state. `proposed_tags` is a living document regenerated by the scanner on every scan (from ML labels, location data, keywords). After reconcile pushes tags to Flickr, the next scan may add new ML labels to `proposed_tags`, making the same photos look "dirty" again. The same photos get reprocessed on every run.

**Bug 2 — no DB update after reconcile `--fix`.**  
When `check_photo()` calls `add_tags()` successfully, nothing is written back to the DB. The local picture stays wrong. On the next scan, `propose_tags()` reads a stale `flickr_tags` cache, may recompute a different `proposed_tags`, and the cycle continues.

---

## Design

### Three semantically distinct tag fields

| Field | Owner | Semantics |
|---|---|---|
| `proposed_tags` | Scanner | Living document. Regenerated from ML labels, location, keywords on every scan. May change freely. |
| `flickr_tags` | Metadata puller | Read-only cache of last observed Flickr state. Only `metadata_puller.py` writes this. |
| `pushed_tags` | BP write operations | Cumulative set of tags BP has confirmed pushed to Flickr. Append-only on success. |

Reconcile's tag check becomes `pushed_tags ⊆ actual_flickr_tags`. New items in `proposed_tags` that haven't been pushed yet are invisible to reconcile — they belong to the push step, not the verify step.

---

### New column: `pushed_tags TEXT`

Added to `photos` via migration 016.

```sql
pushed_tags TEXT  -- JSON array; cumulative set of all tags ever successfully pushed to Flickr
                  -- NULL  = never pushed (no ledger entry exists)
                  -- '[]'  must never be written — use NULL for "nothing pushed"
                  -- non-null = write ledger maintained by BP; append-only on confirmed success
```

**NULL vs `'[]'` distinction:** NULL means BP has never confirmed pushing any tags for this photo. An empty JSON array `'[]'` must never be written — the push sites guard against this by only writing `pushed_tags` when the tag list is non-empty. This keeps query predicates (`WHERE pushed_tags IS NOT NULL`) unambiguous.

---

### Durable desired state — re-addition is intentional

**`pushed_tags` represents BP's desired Flickr tag state, not a historical log.**

If a user manually removes a tag from Flickr, reconcile will detect it as missing and re-add it on the next `--fix` run. This is intentional: BP treats pushed tags as durable desired state and will enforce them on Flickr. Future maintainers should not treat this re-addition as a bug.

To permanently stop BP from managing a specific tag, a user must either:
- Remove the tag from `pushed_tags` in the DB directly, or
- Clear `pushed_tags` to NULL (full reset of the ledger for that photo)

This behaviour is equivalent to a desired-state reconciliation pattern (similar to Kubernetes: if you want "cat" off Flickr, you must remove it from the desired state, not just delete it from Flickr).

### DB write sites — three places push tags, all must write `pushed_tags` on success

**Invariant: Flickr API call first, DB ledger update second. Never the reverse.**  
If the process crashes between the API call and the DB write, the next reconcile run will detect the missing tag and re-push it — safe and convergent. Updating the DB first would falsely mark tags as pushed even if the API call fails.

**1. `poller/poller.py` — `_push_to_flickr()`** (initial push after human review)

After `client.add_tags(flickr_id, tags)` succeeds:
```python
db.conn.execute(
    "UPDATE photos SET tags_pushed_flickr = 1, pushed_tags = ? WHERE flickr_id = ?",
    (json.dumps(sorted(tags)), flickr_id),
)
```
Only written when `tags` is non-empty (existing guard).

**2. `reviewer/app.py` — `api_push_approved`** (bulk push from review UI)

Same pattern: after `add_tags()` succeeds per photo, write `proposed_tags` as `pushed_tags`.

**3. `poller/reconcile.py` — `check_photo()` in `--fix` mode** (additive update)

After `client.add_tags(flickr_id, missing)` succeeds:
```python
current_pushed = json.loads(row.get("pushed_tags") or "[]")
new_pushed = sorted(set(current_pushed) | set(missing))
db.conn.execute(
    "UPDATE photos SET pushed_tags = ? WHERE id = ?",
    (json.dumps(new_pushed), result["row_id"]),
)
db.conn.commit()
```

This is additive — extends the ledger with newly confirmed tags rather than overwriting it.

`check_photo()` signature gains a `db` parameter: `check_photo(client, row, db, fix, verbose)`.

### Reconcile changes

**Query**: add `pushed_tags` to the SELECT.

**Tag check source**: replace `row["proposed_tags"]` with `row["pushed_tags"]`. If `pushed_tags` is NULL or empty, skip the tag check for that photo entirely — there is nothing in the ledger to verify.

Everything else in the tag check (`missing = sorted(expected_tags - flickr_tags)`, the mismatch/fix/error counting) stays unchanged.

### New subcommand: `bp tag-writeback`

New file: `poller/tag_writeback.py`  
Entry point added to `bp` CLI alongside `reconcile`.

**Purpose:** propagate `pushed_tags` back to Photos.app as explicit keywords. Explicit keywords appear in Smart Albums; ML-only labels do not.

**Algorithm:**
1. Query DB: `WHERE pushed_tags IS NOT NULL AND uuid IS NOT NULL` (Photos-linked records only)
2. Open `photoscript.PhotosLibrary()`; look up each photo by UUID
3. Merge: `new_keywords = sorted(set(photo.keywords) | set(pushed_tags))`
4. Write only if `new_keywords != sorted(photo.keywords)`
5. Report per-photo: `ok` | `updated` | `not_found` | `error`

**Options:**
- `--config PATH` (required)
- `--dry-run` (report without writing) — required from day one; Photos.app keyword writes are user-visible
- `--limit N` (default 500)
- `--verbose` (show `ok` results)
- `--source pushed-tags|proposed-tags` (default: `pushed-tags`) — which DB field to read tag candidates from; only `pushed-tags` is implemented initially, but the flag is wired from day one for forward compatibility

**Constraints:**
- Photos.app must be running. Exits early with a clear message if `photoscript` cannot connect.
- Additive only — never removes existing keywords.
- Not part of the daemon; intended for manual or scheduled runs.

---

### Writeback feedback loop analysis

**Is there a self-amplifying growth risk?**

`propose_tags()` sources tags from four inputs, deduplicated into a set:
1. `flickr_tags` (existing Flickr tags)
2. `keywords` (Apple Photos keywords)  ← writeback target
3. `apple_labels` (ML labels)
4. Location tags

If writeback adds "cat" to Photos.app keywords (from ML label), the next scan picks up "cat" via source 2. Since `propose_tags()` uses a set, "cat" appears once in `proposed_tags` regardless of how many sources contribute it. **No amplification for stable ML labels.**

**The bounded risk — ML label drift:**  
If Apple's ML later changes a label (e.g. "automobile" → "vehicle"), LABEL_REMAP maps "automobile" → "car". Writeback has already written "car" to Photos.app keywords. On the next scan, "car" still appears via source 2, AND "vehicle" appears via source 3. `proposed_tags` grows by one tag. "vehicle" will be pushed to Flickr and written back to Photos.app. This is **bounded growth** (one new tag per ML label change), not unbounded amplification.

**Design decision — writeback keywords are intentionally indistinguishable from user-authored keywords.**  
They merge permanently into the photo's keyword set. This is the correct behaviour: once a tag has been pushed to Flickr and written back to Photos.app, it is part of that photo's metadata. Distinguishing BP-authored keywords from user-authored keywords (e.g. via a `bp:` prefix) would add complexity and make the data worse for Smart Albums. The bounded ML-drift risk is acceptable.

### What this does not change

- `proposed_tags` continues to be regenerated by the scanner on every scan. New ML labels still propagate to Flickr via the normal push step — reconcile just won't re-push them on every run.
- `flickr_tags` (metadata puller cache) is unchanged. The puller still owns that field.
- No change to `tags_pushed_flickr` semantics — it remains the boolean "we pushed something."
