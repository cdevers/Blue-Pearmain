# Tags Filter Dimension — Spec (#156)

> Status: approved for implementation

## Overview

Add a tag input to the shared `_filter_bar.html` Jinja macro as a sixth filter dimension, wired to both `/library` and `/map`. The `tag` field enters `normalize_shared_filters()` so both routes normalise it the same way. A new `db.tag_names()` method supplies the datalist values.

---

## Data model

Tags are stored in two JSON array columns on the `photos` table:

- `photos_tags` — Apple Photos keywords; 2,609 distinct values; human-readable (`boston`, `concert`, `family`, …). This is the **datalist source**.
- `flickr_tags` — Flickr tags; 99,256 distinct values; includes machine tags (`exif:orientation=horizontalnormal`, `camera:make=nikoncorporation`, …). **Not used for the datalist.**

The filter logic checks **both** columns for an exact value match — consistent with the existing `_library_where()` `tag` clause. This preserves backwards compatibility with existing `?tag=` URLs and catches photos where a keyword exists on one side but not the other.

**Case sensitivity:** Matching is intentionally case-sensitive — identical to the behaviour of the existing `tag` param in `_library_where()`. Users who accepted the datalist suggestion will always get an exact match. Anyone typing manually needs to match case. A comment in the SQL implementation must document this so it isn't "fixed" unilaterally on only one side.

## New: `db.tag_names()` method

```python
def tag_names(self) -> list[str]:
    """Return distinct tag values from photos_tags JSON arrays, sorted alphabetically.

    Excludes blank/whitespace-only values that can appear when Apple Photos
    exports empty keyword entries.
    """
    rows = self.conn.execute(
        "SELECT DISTINCT j.value "
        "FROM photos p, json_each(p.photos_tags) j "
        "WHERE p.photos_tags IS NOT NULL "
        "  AND p.photos_tags NOT IN ('null', '[]', '') "
        "  AND p.flickr_deleted = 0 "
        "  AND trim(j.value) != '' "
        "ORDER BY j.value"
    ).fetchall()
    return [r["value"] for r in rows]
```

Pattern mirrors `person_names()`. Returns all tags (no minimum-count filter). The `trim(j.value) != ''` guard prevents blank autocomplete entries from `[""]` or `["   "]` arrays.

## Route changes

### `library()` route

- **Remove** `tag = request.args.get("tag") or None` (replaced by `sf["tag"]`).
- **Add** `tag: str | None = sf["tag"] or None` after the `sf` call.
- **Add** `tag_names = db().tag_names()` alongside the existing `person_list` call.
- **Pass** `tag_names=tag_names` to `render_template()`.
- **`filters` dict**: `"tag"` is already present; value comes from `sf["tag"]` now.
- **`filter_count`**: include `(1 if tag else 0)` in the combined counter — tag is a shared dimension.

Note: `filter_count` is intentionally heuristic UI state (one bucket per active dimension) rather than a mathematically exact count. Year range counts as one bucket regardless of whether one or both bounds are set; a future multi-tag selector would similarly count as one. Document this in a comment in `library.html` so future contributors don't try to make the counter exact.

### `api_map_photos()` route

After the person WHERE fragment, add:

```python
# Tag — case-sensitive exact match in either photos_tags or flickr_tags.
# Intentionally case-sensitive for consistency with _library_where() tag logic;
# datalist suggestions always produce exact-case values so this is correct for
# normal usage. Do not add LOWER() without also changing _library_where().
map_tag = (sf["tag"] or "").strip()
if map_tag:
    where_frags.append(
        "(EXISTS (SELECT 1 FROM json_each(p.flickr_tags) WHERE value = ?) "
        "OR EXISTS (SELECT 1 FROM json_each(p.photos_tags) WHERE value = ?))"
    )
    where_params.extend([map_tag, map_tag])
```

**Tech debt note:** The tag EXISTS clause now appears in both `_library_where()` and `api_map_photos()`. A future `tag_filter_clause(alias, value) -> tuple[str, list]` helper would consolidate this. Not required for this PR.

### `map_view()` route

- **Add** `tag_names = db().tag_names()` alongside other DB calls.
- **Add** `"tag": sf["tag"]` to the `initial_filters` dict.
- **Pass** `tag_names=tag_names` to `render_template()`.

## Macro changes — `_filter_bar.html`

Updated signature:

```jinja
{% macro filter_bar(albums, person_names, filters,
                    datalist_id="shared-persons",
                    tag_names=None,
                    tag_datalist_id="shared-tags") %}
```

`tag_names=None` (not `[]`) avoids the mutable-default visual smell. Inside the macro, iterate with `{% for t in (tag_names or []) %}`.

New tag input row (after person, before privacy):

```html
<label>Tag
  <input type="text" name="tag" value="{{ filters.tag or '' }}"
         list="{{ tag_datalist_id }}" placeholder="tag…" style="width:160px">
  <datalist id="{{ tag_datalist_id }}">
    {% for t in (tag_names or []) %}
    <option value="{{ t | e }}">
    {% endfor %}
  </datalist>
</label>
```

**Existing callers remain valid** — `tag_names` defaults to `None` (empty datalist rendered, input still works for manual entry) and `tag_datalist_id` has a safe default.

**Datalist scale note:** 2,609 entries is within browser tolerance today. If this grows significantly, future mitigations include a minimum-count threshold, prefix-based async autocomplete, or a capped list. No immediate action required.

## Testing

### `db.tag_names()`
- Returns a sorted list of strings.
- Excludes photos with `flickr_deleted = 1`.
- Returns `[]` when no photos have `photos_tags`.
- Returns `[]` (not `[""]`) when `photos_tags` contains only blank entries like `[""]` or `["   "]`.

### `normalize_shared_filters()` — tag field
- `?tag=boston` → `sf["tag"] == "boston"`.
- `?tag=+boston+` → `sf["tag"] == "boston"` (whitespace stripped).
- `?tag=` → `sf["tag"] == ""`.
- `?tag=%20boston%20` → `sf["tag"] == "boston"` (URL-decoded then stripped).

### `library()` with `?tag=`
- `?tag=boston` returns only photos where `boston` appears in `photos_tags` or `flickr_tags`.
- `?tag=` (empty) returns unfiltered results.
- `?tag=no-such-tag` returns 0 photos.
- `?tag=Boston` (capital B) returns 0 photos when only `boston` (lowercase) exists — case-sensitive by design.
- Filter chip appears when `tag` is non-empty.

### `api_map_photos()` with `?tag=`
- `?tag=boston` scopes dots/trail to photos tagged `boston`.
- `?tag=` (empty) returns all photos (subject to other filters).

### `map_view()` deep-link
- `?tag=boston` → `initial_filters["tag"] == "boston"` → macro pre-fills input → panel auto-opens.

### Cross-page nav
- Library "View on map" link with active `tag` filter carries `?tag=boston` to `/map`.
- Map `openInLibrary()` with active `tag` filter carries `?tag=boston` to `/library`.

### NULL-column combinations
Both tag columns may be NULL independently. Verify:

| `photos_tags`   | `flickr_tags`   | `?tag=boston` | expected |
|-----------------|-----------------|---------------|----------|
| `["boston"]`    | NULL            | match         | 1 photo  |
| NULL            | `["boston"]`    | match         | 1 photo  |
| `["boston"]`    | `["boston"]`    | match         | 1 photo (no duplicate) |
| `[]`            | `["boston"]`    | match         | 1 photo  |
| `["boston"]`    | `[]`            | match         | 1 photo  |
| NULL            | NULL            | no match      | 0 photos |

The `EXISTS (SELECT 1 FROM json_each(...))` pattern returns no rows when the column is NULL, so these cases are handled correctly by SQLite — but test them explicitly since JSON null-handling is subtle.

### Whitespace roundtrip
`?tag=%20boston%20` normalises identically in both `/library` and `/api/map-photos` — both use `normalize_shared_filters()`, so this is guaranteed structurally. Test confirms the guarantee holds.
