# Privacy Policy Precedence

BP assigns every photo a `privacy_state` through two separate layers: the **classifier** (run at scan time) and the **proposal/tag-protection system** (run when syncing metadata from Flickr). This document describes the rules at each layer and how they interact.

---

## Classification layer (`analyzer/privacy.py`)

The `classify()` function is called whenever a photo is scanned or re-scanned. Rules are evaluated top-to-bottom; the first match wins. Photos whose `privacy_state` has already been set by a human review decision are not re-classified (see **Decision lock** below).

| Priority | Rule | Trigger | Result |
|----------|------|---------|--------|
| 1 | **Apple home flag** | `place_ishome = true` in Apple Photos metadata | `auto_private` |
| 2 | **Geofence zone — auto_private policy** | Photo GPS is within a zone with `policy = auto_private` | `auto_private` |
| 2 | **Geofence zone — flag_review policy** | Photo GPS is within a zone with `policy = flag_review` | `needs_review` |
| 2 | **Geofence zone — auto_public policy** | Photo GPS is within a zone with `policy = auto_public` | *(falls through)* |
| 3 | **Person policy — always_private** | A named person in the photo matches a `person_policies` entry with `policy = always_private` (case-insensitive) | `auto_private` |
| 4 | **Named persons (not self)** | Apple Photos has identified named people other than the photographer | `needs_review` |
| 5 | **Unknown faces** | Unidentified faces detected above quality threshold | `needs_review` |
| 6 | **People/crowd labels** | Apple's scene labels include people-indicating terms (`person`, `crowd`, `audience`, etc.) | `needs_review` |
| 7 | **Human body detection** | `media_analysis` confidence exceeds threshold | `needs_review` |
| — | **Default** | No people signals detected | `candidate_public` |

### Key notes

- **Home flag beats everything.** Apple's `ishome` flag is the unconditional override, set based on your Home address in Apple Maps. No geofence or policy overrides it.
- **Geofence zones are checked in DB order** for the first matching zone. If a photo falls inside multiple zones, the zone returned first by the DB query wins. In practice, zones should not overlap for `auto_private`/`flag_review`; `auto_public` zones are intended to carve out exceptions inside a larger `auto_private` zone.
- **Person policies are checked before general face detection.** A photo with an `always_private` person goes straight to `auto_private` — it never enters the review queue. This is intentional: the policy is meant to suppress these photos permanently without operator intervention.
- **Self-name exclusion.** The photographer's own name (configured via `self_name` in config) is excluded from rules 3 and 4. A selfie does not trigger `needs_review` on its own.
- **Confidence thresholds.** Unknown face detection has a quality floor (`FACE_QUALITY_THRESHOLD`) to filter ghost detections. Human body detection has a confidence floor (`HUMAN_CONFIDENCE_THRESHOLD`). Both are constants in `analyzer/privacy.py`.

## Decision lock

Once a human records a review decision via the review UI, the photo's `privacy_state` is **permanently protected** from re-classification. `record_review()` sets `review_decision` and stamps `reviewed_at`; all background scan/poll operations check for a non-null `review_decision` and skip re-classification if found.

This means:

- `bp scan`, `bp poll`, and `bp reconcile` will update metadata (GPS, tags, labels) but will never revert a human decision.
- The only way to change a reviewed photo's state is via the UI (including the undo function, which restores the previous `review_decision`).

---

## Tag protection layer (`db/db.py` → `supersede_managed_tag_proposals`)

Separate from privacy classification, BP's metadata sync system tracks which tags on Flickr were machine-generated by BP (location tags, label tags — stored in `proposed_tags`). When Flickr tags are synced back and compared against Apple Photos keywords, a divergence proposal would normally be raised for every BP-managed tag that isn't yet in Photos.

The tag protection rule (`bp prune-proposals`, also run automatically via `bp pipeline`) **supersedes** any pending `flickr→photos` tag proposal whose net diff consists entirely of BP-managed tags. This prevents BP from proposing to write its own machine-generated tags back into Apple Photos as human-entered keywords.

This rule operates on **proposals**, not on `privacy_state` — it is independent of the classification layer above.

---

## Export

`bp export` snapshots the current `privacy_state` of each photo as recorded in the DB at export time. It does not re-classify. The export reflects whatever combination of scan results, policies, and human decisions produced the current state — it is not a re-derivation from first principles.

---

## Adding new rules

When adding a new privacy rule:

1. Determine where it belongs in the precedence table above.
2. If it overrides human decisions, it goes in the classification layer but the decision-lock check must be considered carefully — adding a rule that fires after the decision lock would have no effect on reviewed photos.
3. If it operates on proposals, it belongs in the proposal layer alongside `supersede_managed_tag_proposals`.
4. Update this document.
